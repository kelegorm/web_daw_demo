**Step A: Add facade interfaces to `src/engine/types.ts`**

Keep all existing exports (AudioModule, MODULE_KINDS, ModuleKind, MeterFrame, MeterSource). Add three new interfaces:

1. `TrackFacade` interface:
   - `readonly meterSource: MeterSource`
   - `setGain(db: number): void`
   - `getGain(): number`
   - `setMute(muted: boolean): void`
   - `isMuted(): boolean`
   - `dispose(): void`

   These map to the existing `TrackStripGraph` methods (`setTrackVolume`, `trackVolume`, `setTrackMuted`, `isTrackMuted`, `meterSource`). The rename to `setGain/getGain/setMute/isMuted` is intentional — the facade uses domain-appropriate names, not implementation names.

2. `MasterFacade` interface:
   - `readonly meterSource: MeterSource`
   - `setGain(db: number): void`
   - `getGain(): number`

   Maps to `MasterStripGraph` methods (`setMasterVolume`, `masterVolume`, `meterSource`). No mute on master.

3. `DeviceFacade` interface:
   - `readonly kind: string`
   - `readonly trackId: string`
   - `getParam(name: string): number | boolean`
   - `setParam(name: string, value: number | boolean): void`

   This is a generic interface for panner, limiter, synth etc. — it provides a uniform param get/set surface. Implementation is NOT in this phase; only the interface definition.

**Step B: Create `src/engine/engineSingleton.ts`**

This is the NEW engine module. It does NOT modify or delete `audioEngine.ts` — that file stays as-is for now.

Module structure:

```typescript
import { createLimiter, type LimiterGraph } from '../hooks/useLimiter';
import { createMasterStrip, type MasterStripGraph } from '../hooks/useMasterStrip';
import { createTrackStrip, type TrackStripGraph } from '../hooks/useTrackStrip';
import { createMeterSource } from './meterSource';
import type { MasterFacade, TrackFacade, MeterSource as MeterSourceType } from './types';

// Use Tone.js AudioContext — see AudioContext ownership note below
import * as Tone from 'tone';

const DEFAULT_TRACK_ID = 'track-1';

interface EngineApi {
  getTrackFacade(trackId: string): TrackFacade;
  getMasterFacade(): MasterFacade;
  createTrackSubgraph(trackId: string): TrackFacade;
  removeTrackSubgraph(trackId: string): void;
  /** Limiter input meter (pre-limiter signal level) */
  getLimiterInputMeter(): MeterSourceType;
  /** Limiter gain reduction in dB (0 = no reduction) */
  getLimiterReductionDb(): number;
  // Legacy accessors for backward compat during migration (Plans 01-03 and Phase 2-3)
  _legacy: {
    readonly audioContext: AudioContext;
    readonly limiterGraph: LimiterGraph;
  };
}

let _engine: EngineApi | null = null;

export function getAudioEngine(): EngineApi {
  if (!_engine) {
    _engine = createEngineInternal();
  }
  return _engine;
}

export function _resetEngineForTesting(): void {
  _engine = null;
}
```

Internal implementation (`createEngineInternal()`):

1. **AudioContext:** Get from Tone.js: `const audioContext = Tone.getContext().rawContext as AudioContext;`

   **AudioContext ownership decision (resolves RESEARCH.md Open Question #3):**
   The singleton MUST use Tone.js's AudioContext, not create its own via `new AudioContext()`.
   Reason: Plan 01-03 wires the Tone.js synth output into the track strip input within
   the same Web Audio graph. Web Audio API prohibits `connect()` calls across different
   AudioContext instances — it throws an InvalidAccessError. Since the existing synth
   (`createToneSynth`) and transport (`Tone.getTransport()`) are bound to Tone's context,
   all engine nodes (GainNodes, AnalyserNodes, DynamicsCompressorNode, etc.) must live on
   the same context.

   The RESEARCH.md Anti-Patterns entry about avoiding `Tone.getContext().rawContext` was
   written with a future assumption that devices would be fully decoupled. That assumption
   does not hold for Phase 1: the synth device is still directly wired to the track strip
   in App.tsx. Once devices are fully managed by the engine (Phase 2+), AudioContext
   ownership can be revisited — but for now, sharing Tone's context is the only correct
   approach.

   Do NOT create `new AudioContext()`. Do NOT call `Tone.setContext()` with a custom context
   (this can break Tone.js internals). Simply read Tone's existing context.

2. **preLimiterBus:** `const preLimiterBus = audioContext.createGain(); preLimiterBus.gain.value = 1.0;`
   Unity gain, pass-through summing node.

3. **Master chain:** Create limiter and master strip using existing factories:
   ```
   const limiterGraph = createLimiter(audioContext);
   const masterStripGraph = createMasterStrip(audioContext);
   preLimiterBus.connect(limiterGraph.input);
   limiterGraph.output.connect(masterStripGraph.input);
   masterStripGraph.output.connect(audioContext.destination);
   ```

4. **MasterFacade:** Create an object implementing MasterFacade:
   ```
   const masterFacade: MasterFacade = {
     get meterSource() { return masterStripGraph.meterSource; },
     setGain(db) { masterStripGraph.setMasterVolume(db); },
     getGain() { return masterStripGraph.masterVolume; },
   };
   ```

5. **Track registry:** `const tracks = new Map<string, { facade: TrackFacade; strip: TrackStripGraph }>();`
   Do NOT create the default track in this plan — Plan 01-02 adds `createTrackSubgraph` which will bootstrap the default track.

6. **Stub track methods:** For now, `getTrackFacade`, `createTrackSubgraph`, and `removeTrackSubgraph` should throw `new Error('[engine] track API not yet implemented — see Plan 01-02')`. This keeps the type contract honest while the implementation comes in Plan 02.

7. **Return the EngineApi** with masterFacade, limiter meter access, and the stub track methods. Include `_legacy` accessor for migration compatibility:
   ```
   _legacy: {
     get audioContext() { return audioContext; },
     get limiterGraph() { return limiterGraph; },
   }
   ```

Important constraints:
- Do NOT import from `audioGraphPlan.ts` or the old `audioEngine.ts` — this is a clean break.
- Do NOT create a `dispose()` method — per CONTEXT.md, engine is app-lifetime.
- Keep Tone.js import to the single `Tone.getContext()` call for AudioContext retrieval.
