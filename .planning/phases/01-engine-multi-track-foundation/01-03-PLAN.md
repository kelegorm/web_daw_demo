<task type="auto">
  <name>Task 1: Rewrite App.tsx to use getAudioEngine() singleton and remove all APP_* constants</name>
  <files>src/App.tsx</files>
  <action>
**Understanding the current state:**

App.tsx currently:
1. Calls `useAudioEngine()` which returns `AudioEngine | null` (null while loading)
2. Has `AppWithEngine` component that receives the engine and uses APP_* constants to look up modules
3. Uses `buildUiRuntime()` to map the UI plan into runtime models
4. Hardcodes APP_TRACK_STRIP_ID comparisons for the "is this the primary track?" branching

**Migration strategy:**

The singleton engine is always available (no null state, no loading). The old `AudioEngine` interface with `getSynth(id)`, `getPanner(id)`, etc. still works ‚Äî it's the underlying engine that the singleton wraps. But the singleton wraps it differently.

For this migration, we need to keep the app FUNCTIONALLY IDENTICAL. The key insight: the singleton exposes `_legacy` accessors that give us the underlying graph objects. We can also keep using the old `createAudioEngine` internally ‚Äî the singleton already creates the master chain using the same factories.

But actually, App.tsx currently uses the old engine's `getSynth`, `getPanner`, `getTrackStrip`, `getLimiter`, `getMasterStrip` methods. The new singleton does NOT expose those methods (it exposes `getTrackFacade`, `getMasterFacade`). So we need a bridge.

**Approach:** The singleton needs to internally create the old-style AudioEngine for backward compat OR App.tsx needs to access the underlying modules differently.

Simplest path: The singleton's `_legacy` provides the audioContext, limiterGraph, and trackStripGraph. For synth and panner, App.tsx still needs them. Since the old `createAudioEngine` created synth+panner as part of the single-track plan, and the new singleton does NOT create synth or panner (those are devices, Phase 2+), we have a gap.

**Resolution:** For THIS migration step, keep the old engine creation for synth/panner within App.tsx but access track strip and master strip through the new singleton. This is the incremental approach:

1. **Remove `useAudioEngine()` usage** ‚Äî App.tsx no longer calls `useAudioEngine()` and no longer has the null-engine loading state.

2. **Get the singleton engine at module level:**
   ```typescript
   import { getAudioEngine, DEFAULT_TRACK_ID } from './engine/engineSingleton';
   ```

3. **Keep synth/panner creation in App.tsx for now** ‚Äî the old engine's `createToneSynth()` and `createPanner()` still need to be called because the singleton doesn't manage devices yet. Import them directly:
   ```typescript
   import { createToneSynth } from './hooks/useToneSynth';
   import { createPanner } from './hooks/usePanner';
   ```
   Create them at module level or in the component using the singleton's audioContext.

4. **Replace APP_TRACK_STRIP_ID usage** ‚Äî Instead of `audioEngine.getTrackStrip(APP_TRACK_STRIP_ID)`, use `getAudioEngine()._legacy.getTrackStripGraph(DEFAULT_TRACK_ID)` which returns the TrackStripGraph. The hooks (`useTrackStrip`, `useMasterStrip`, etc.) receive the graph's hook interface, so pass it through.

5. **Replace APP_MASTER_STRIP_ID usage** ‚Äî Use `getAudioEngine().getMasterFacade()` for master volume and meter. But `useMasterStrip` hook expects a `MasterStripHook` ‚Äî the MasterFacade has different method names (setGain vs setMasterVolume). Create an adapter:
   ```typescript
   const masterFacade = engine.getMasterFacade();
   const masterStripAdapter: MasterStripHook = {
     get masterVolume() { return masterFacade.getGain(); },
     setMasterVolume: (db) => masterFacade.setGain(db),
     get meterSource() { return masterFacade.meterSource; },
   };
   ```

6. **Replace APP_LIMITER_MODULE_ID usage** ‚Äî Use `engine._legacy.limiterGraph` which is the LimiterGraph. Create the limiter hook interface from it (same as what `createAudioEngine` did internally).

7. **Replace APP_SYNTH_MODULE_ID and APP_PANNER_MODULE_ID** ‚Äî Use directly-created synth and panner. Connect synth output to panner, panner output to track strip input. This replicates the old engine's wiring but explicitly in App.tsx.

8. **Remove the `if (!audioEngine) return <div id="app" />` null check** ‚Äî the singleton is always available.

9. **Flatten AppWithEngine back into App** ‚Äî no need for the split since there's no loading state.

10. **Remove all APP_* constant definitions** and the `findDeviceModuleIdByKindOrThrow` helper.

11. **Remove `INITIAL_TRACK_PLAN` lookup** ‚Äî use DEFAULT_TRACK_ID instead.

12. **Update the `runtimeTrack.trackStripId === APP_TRACK_STRIP_ID` branches** ‚Äî these become comparisons against DEFAULT_TRACK_ID on the trackId field, or better: since there's only one track right now, all tracks use the facade. Simplify the branching to always use the facade/strip directly.

13. **Update the devicePanelModel mapping** ‚Äî instead of comparing `device.moduleId === APP_SYNTH_MODULE_ID`, compare against the known synth/panner module IDs from the UI plan, or restructure to use the directly-created synth/panner instances.

**Key wiring that must be preserved:**
- Synth output -> panner input -> panner output -> track strip input (the device chain into the track)
- Track strip output -> preLimiterBus (already wired by singleton)
- preLimiterBus -> limiter -> master strip -> destination (already wired by singleton)
- Transport controller receives synth and track strip hooks
- MIDI keyboard receives synth hook

**What to delete:**
- `import { useAudioEngine } from './hooks/useAudioEngine'`
- All five `APP_*` constant definitions
- `findDeviceModuleIdByKindOrThrow` function
- `INITIAL_TRACK_PLAN` lookup
- The `AppWithEngine` component split (flatten back to `App`)
- The null-engine loading guard

**What to keep/update:**
- All `use*` hook calls (useToneSynth, usePanner, useTrackStrip, useMasterStrip, useLimiter) ‚Äî they receive adapted inputs from the singleton
- `buildUiRuntime` call ‚Äî it still needs an AudioEngine-shaped object; either pass the old engine from singleton's internal creation, or construct a compatible adapter
- All component rendering (Toolbar, TrackZone, DevicePanel, MidiKeyboard) ‚Äî identical

**IMPORTANT: Do not break the UI.** The synth/panner/limiter device chain must still work. If bridging is complex, it's acceptable to keep a thin wrapper that creates the old AudioEngine internally but sources its AudioContext from the singleton. The critical requirement is: APP_* constants are gone, useAudioEngine.ts is deleted, getAudioEngine() is the entry point.

**Alternative simpler approach if the above is too complex:**

Keep the old `createAudioEngine` call in App.tsx but source its AudioContext from the singleton to prevent duplicate contexts. Replace the `useAudioEngine()` hook with a direct call. Delete the constants by replacing them with direct references from the old engine's plan IDs (which are already exported as `DEFAULT_PLAN_SYNTH_ID` etc. from audioGraphPlan.ts). This is less elegant but guarantees no behavioral change.

In this case:
```typescript
import { getAudioEngine } from './engine/engineSingleton';
import { createAudioEngine, DEFAULT_AUDIO_MODULE_FACTORY_MAP } from './engine/audioEngine';
import { DEFAULT_AUDIO_GRAPH_PLAN, DEFAULT_PLAN_SYNTH_ID, DEFAULT_PLAN_PANNER_ID, DEFAULT_PLAN_TRACK_STRIP_ID, DEFAULT_PLAN_LIMITER_ID, DEFAULT_PLAN_MASTER_STRIP_ID } from './engine/audioGraphPlan';

// Ensure singleton is initialized (creates AudioContext)
const engine = getAudioEngine();

// Create old-style engine using same AudioContext for device chain compat
const oldEngine = createAudioEngine(DEFAULT_AUDIO_GRAPH_PLAN, DEFAULT_AUDIO_MODULE_FACTORY_MAP);

function App() {
  // No loading state needed ‚Äî engines are module-level
  const toneSynth = useToneSynth(oldEngine.getSynth(DEFAULT_PLAN_SYNTH_ID));
  // ... etc, using DEFAULT_PLAN_*_ID instead of APP_* constants
}
```

This approach still deletes:
- All APP_* constants (replaced by DEFAULT_PLAN_*_ID imports)
- useAudioEngine.ts
- The null guard / AppWithEngine split
- findDeviceModuleIdByKindOrThrow

And introduces getAudioEngine() as the canonical entry point. The old engine is kept temporarily for device chain compat ‚Äî it gets removed in Phase 3 (App.tsx Teardown).

**Choose whichever approach produces the cleanest, most reliable result.** The simpler alternative is preferred if the full migration creates too many moving parts for one plan.
